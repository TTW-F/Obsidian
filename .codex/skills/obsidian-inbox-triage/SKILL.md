# Obsidian Inbox Triage

把 `00-收集` 里的内容快速分流，不追求一次写完，只追求让每条内容拥有更清晰的下一步。

优先处理“是否值得保留、该去哪里、还缺什么”三个问题，再做轻量修整。

## 工作流

1. 先确认本次处理范围。
   典型范围是单篇收集笔记、某个批次的新笔记，或整个 `00-收集` 目录。
2. 判断每条内容的状态。
   只在以下四类里选一类，不要摇摆：
   - 删除：没有复用价值、已过时、纯噪音。
   - 合并：主题已经存在，只需要把有效片段并入旧笔记。
   - 升级：值得成为正式笔记，移动到 `10-领域` 或 `20-主题`。
   - 暂存：内容有价值，但信息不足，先改造成更易后续处理的中间态笔记。
3. 给出最小必要整理。
   包括改标题、拆分混杂主题、补一两句摘要、补 `tags`/`type`、加一个明确的下一步。
4. 执行分流。
   需要移动时，按库内约定落到正确目录；需要合并时，优先保留信息密度更高的原笔记。
5. 汇报结果。
   说明处理了哪些文件、每条内容被如何处置、哪些条目仍需你亲自判断。

## 分流规则

先读 `references/triage-playbook.md`，再执行。

使用以下判断标准：

- 放到 `10-领域`：长期维护的工具、流程、方法、个人能力栈。
- 放到 `20-主题`：可抽象复用的概念、知识点、范式、题型。
- 放到 `30-地图`：用于导航、串联、总览，不承载大段细节。
- 保留在 `00-收集`：只有在暂时无法判断去向，但未来大概率还会处理时。

如果一篇收集笔记同时包含多个主题，先拆，再分流，不要整篇硬塞进某个目录。

## 输出要求

完成任务时，尽量给出：

- 已删除、已合并、已升级、已暂存的清单。
- 每篇升级后笔记的目标目录。
- 仍然模糊的条目和原因。
- 下一轮最值得处理的 1 到 3 个入口。

## 边界

- 不要因为内容粗糙就直接删除；先判断是否存在未来复用价值。
- 不要为了“看起来整洁”过度改写原意。
- 不要把地图笔记写成资料堆砌。
- 如果目标笔记已经存在，优先合并，不要平白制造近义重复笔记。

## 常见触发语

- “帮我清一下 00-收集”
- “把这批 inbox 笔记分流”
- “看哪些该删，哪些该升级为正式笔记”
- “把零散记录整理到我的知识库结构里”
