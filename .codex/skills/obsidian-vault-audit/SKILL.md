# Obsidian Vault Audit

把知识库当成持续演化的系统来巡检，找出最影响可维护性的少数问题。

先找高杠杆问题，再做小修，不要陷入机械式全量美化。

## 巡检顺序

1. 看结构。
   检查 `00/10/20/30/90` 分层是否仍然清晰，有没有明显放错位置的笔记。
2. 看命名。
   找泛标题、重复标题风格、难检索标题。
3. 看元数据。
   检查 frontmatter 是否混乱、缺失或字段失控。
4. 看链接与导航。
   找孤立的重要笔记、应进入地图却未进入的笔记、失效或薄弱入口。
5. 看重复与重叠。
   找可能需要合并的近义笔记、过度拆分的系列笔记。
6. 看文档准入。
   找那些本质上更像 agent 分析报告、研究过程记录、框架安置说明，而不是正式知识笔记的内容。

## 输出方式

先读 `references/audit-checklist.md`。

优先给出按影响排序的问题清单：

- 高优先级：会明显影响后续检索、复用、导航。
- 中优先级：会降低一致性，但暂不阻塞使用。
- 低优先级：局部细节优化。

如果用户希望你直接改，优先执行低风险修复：

- 标题微调
- frontmatter 清理
- 补相关链接
- 小范围移动到更合适目录
- 删除或压缩明显属于临时分析的铺垫段落

## 关于“分析冒充知识”的巡检规则

把以下情况视为重点问题：

- 观察、推断、结论混写，读者无法分辨哪些是事实，哪些是判断。
- 文章主要在安置研究框架、目录边界、站位关系，而不是沉淀可复用知识。
- 语气像在对外讲解主题，而不是在为未来检索和复用服务。
- “意义、定位、方法论”过多，具体证据、例子、用法和坑点过少。
- 笔记读起来不像“一个人写给未来自己的笔记”，而像“agent 整理过的一份说明”。

遇到这类问题时，不要只说“风格怪”。
要指出它到底更像哪一类错误文档：

- 临时分析
- 研究计划
- 对外说明
- 空泛总论

## 结果要求

完成任务时，尽量给出：

- 发现了哪些系统性问题。
- 哪些是立刻值得修的前 3 项。
- 哪些改动已经直接执行。
- 哪些建议先观察，不要急着动。

## 边界

- 不要把“风格不同”误判成“必须统一”。
- 不要为了巡检而大规模重写内容。
- 不要在没有足够证据时合并两篇概念近似但边界不同的笔记。
- 不要把所有问题都列成同样优先级。
- 不要把真实有价值的个人判断误判为“主观废话”；要区分“个人理解”与“临时分析腔”。
