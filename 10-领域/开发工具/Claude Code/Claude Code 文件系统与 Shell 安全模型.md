# Claude Code 文件系统与 Shell 安全模型

## 这是什么

这篇笔记记录 Claude Code 如何为“读写文件、执行命令、调用本地环境能力”建立可控边界。

这不是某个单独工具的附属规则，而是整套 agent 运行时的基础设施。只要 agent 能直接改文件、跑命令、接触本地环境，它就必须回答：哪些操作允许做、在哪些范围里做，以及出问题时靠什么兜底。

## 为什么重要

- 对 coding agent 来说，执行能力本身并不稀缺，真正稀缺的是可控执行
- 一旦系统具备文件读写、Shell / PowerShell、外部工具调用等能力，就天然进入高风险区
- 这套模型解决的不是“能不能执行”，而是“如何在不失去可用性的前提下，让执行保持可控、可解释、可隔离”

## Claude Code 如何给执行能力加护栏

### 1. 工具级边界

不同能力的风险不同，治理方式也不应该一样。

文件读取、文件写入、Bash、PowerShell、子代理调度虽然都属于“执行”，但风险来源并不相同。把它们拆开建模，意味着系统不会用一套过于粗糙的规则覆盖所有场景。

### 2. 规则级边界

权限控制不是简单的 allow / deny 二选一，而是继续细化到更具体的约束条件。

常见约束会落在：

- 工具类型
- 命令前缀
- 可访问目录
- 执行模式
- 是否命中高风险规则

这层边界的作用，是让系统既能拦截危险操作，又不至于把真实工作都堵死。

### 3. Shell 语义级边界

Shell 安全不能只靠字符串黑名单，因为命令风险往往和执行语义有关。

一个更成熟的模型，通常还会继续判断：

- 当前默认 shell 是什么
- 命令是否符合允许的前缀模式
- 操作是否只读
- PowerShell 是否触发特有的高风险 cmdlet 或组合方式

这说明 Claude Code 处理的不是“文本命令”，而是“带执行语义的命令环境”。

### 4. 环境隔离边界

权限规则只能解决“能不能做”，环境隔离解决的是“即使做了，影响能不能被限制住”。

像 sandbox、worktree、remote isolation 这类隔离机制，核心价值在于缩小执行副作用，减少：

- 文件误改后影响整个工作区
- 多任务共享环境导致状态污染
- 子代理或远端执行互相覆盖上下文和结果

## 一个具体场景怎么理解这套模型

如果 agent 需要在工作区里运行命令、生成补丁并调用子代理继续处理，那么系统至少要同时回答：

- 当前命令是否属于允许范围
- 写操作是否只落在指定目录
- PowerShell / Bash 风险规则是否命中
- 子代理是否还能继续继承这些边界
- 即使出错，影响会不会被隔离在 worktree 或沙箱里

这个场景能帮助我记住：文件系统安全和 Shell 安全不是单点限制，而是一整套执行边界治理。

## 最该记住的点

- Claude Code 会把文件读写、Bash、PowerShell、子代理调度拆开建模，而不是用一套粗规则覆盖所有执行能力
- PowerShell 不能简单复用 Bash 的风险模型
- 只靠权限弹窗不够，危险识别最好发生在真正执行之前
- 完整护栏通常至少包含规则约束、命令语义理解和执行环境隔离三层

## 易错点

- 容易把 Shell 安全只理解成“命令黑名单”
- 容易把权限规则当成唯一护栏，而忽视 worktree、sandbox、remote isolation 等隔离机制
- 容易假设 Bash 和 PowerShell 可以共用一套判断标准
- 容易只盯着“是否允许执行”，忽略副作用是否可收敛、可回退、可隔离
- 容易忽视 PowerShell 在 Claude Code 里是单独处理的对象，结果误以为所有 shell 风险都能复用同一套判断逻辑
