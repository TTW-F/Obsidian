# Claude Code Compact、History Snip 与长上下文收缩

## 这是什么

这篇笔记记录 Claude Code 怎样在长会话里管理历史：什么时候压缩旧上下文、怎样裁切历史边界，以及怎样在上下文受限时仍然维持任务连续性。

这里的重点不是“省 token”，而是“长任务怎样在有限上下文里继续稳定工作”。

## 为什么重要

- 面向长任务的 agent 必然会遇到上下文窗口被历史吃满的问题
- 历史越长，请求越贵，也越容易把当前任务主线淹没
- 旧信息不能全丢，但也不能每轮都原样重放，所以历史收缩是运行时基础能力，不是体验优化

## 这层真正要平衡什么

可以先把它理解成在三个目标之间做平衡：

- 保住任务连续性
- 控制上下文成本
- 降低用户对“历史被裁掉”的感知

真正难的地方，不是把消息删短，而是压缩之后系统仍然知道自己在做什么、为什么做到这里、接下来该接哪一步。

## Compact 和 History Snip 可以先怎样理解

### Compact

`Compact` 更像对会话历史做整体压缩，让旧历史不再以原始、完整消息序列继续参与后续推理。

### History Snip

`History Snip` 更像带边界感的历史裁切。

它强调的不是简单摘要，而是：

- 找到适合裁切的历史边界
- 保留继续执行所需的关键结构
- 让后续运行时还能基于压缩后的历史继续工作

所以这层更接近“上下文内存管理”，而不是普通聊天产品里的历史清理。

## 为什么这层不能只靠人工处理

长会话 agent 不能依赖“用户自己决定什么时候删历史”，因为：

- 用户未必知道哪些历史已经安全可裁
- 运行时比用户更清楚当前需要保留哪些结构
- 一旦压缩时机和力度不稳定，体验就会明显波动

所以历史收缩必须是运行时机制，而不是手动维护。

## 一个具体场景怎么理解这层

假设一个会话已经持续了很久，前面做过很多文件读取、工具调用和中间决策。

这时系统如果还把所有历史完整带着，就会同时遇到：

- 成本越来越高
- 当前主线越来越不清楚
- 新一轮推理被陈旧细节淹没

而如果直接粗暴删掉历史，又会让后续执行像突然失忆。

这正是 Compact 和 Snip 存在的原因：它们试图把“完整历史”变成“仍然可工作的压缩历史”。

## 最该记住的点

- 长会话 agent 不能依赖“把完整历史一直带着”
- 历史收缩必须是运行时机制
- 压缩历史时，真正难的是保留任务结构，而不是减少 token 数
- Compact、Snip、Replay 这类能力通常要配套设计，否则历史一旦被裁切，后续执行容易失稳

## 易错点

- 容易把 compact 理解成“删掉旧消息”
- 容易把上下文压缩只看成省 token
- 容易把 History Snip 当成普通摘要
- 容易忽视触发时机和裁切力度，实际它们会直接影响稳定性和体验
- 容易把这篇笔记读成抽象的“长上下文哲学”，而忽略 Claude Code 这里实际在处理的是 `Compact`、`History Snip`、`Replay` 这些配套机制怎样一起维持会话连续性
