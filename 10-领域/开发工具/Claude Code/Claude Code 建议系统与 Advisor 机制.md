# Claude Code 建议系统与 Advisor 机制

## 这是什么

这篇笔记记录 Claude Code 怎样在运行中给用户提供建议、提示和下一步引导。

这里的重点不是“有没有提示文案”，而是“复杂能力怎样在真实使用过程中被动态引出来”。

## 为什么重要

- 一个能力很多的 agent 产品，最大风险之一就是用户不知道下一步该怎么用
- 即使功能很强，如果当前可用能力无法被及时发现，体验仍然会像黑盒
- 建议系统本质上是在降低复杂度摩擦，让能力真正变得可用

## 关键线索

从源码命名和入口组织上，Claude Code 里至少存在几类建议型能力：

- advisor 相关逻辑
- prompt suggestion
- tips / example commands
- command suggestions
- skill usage tracking / skill 提示

这些线索说明它不是纯靠外部文档引导，而是在运行时动态帮助用户。

## Advisor 更像在做什么

可以先把 advisor 理解成一种“轻量引导层”。

它的目标不是替代主代理，而是帮助用户：

- 更快进入正确操作路径
- 发现适合当前场景的命令或能力
- 在当前上下文下知道“接下来最值得尝试什么”

这类机制在具备大量命令、工具、plugins、skills 的系统里尤其重要。

## 为什么复杂 agent 产品需要这层

Claude Code 的复杂度不只是模型能力带来的，还来自：

- slash commands 很多
- tools 很多
- 插件和 skills 会继续扩展
- 不同模式下可用能力还会变化

如果没有建议层，用户面对的就会是一整套强能力黑盒。

## 一个具体场景怎么理解建议系统

假设用户正在一个代码仓库里操作，但还没明确知道应该用命令、工具还是某个 skill。

这时一个有用的建议系统 ideally 会根据当前上下文给出：

- 当前最适合的命令入口
- 哪类能力当前环境可用
- 某个 skill 或 workflow 是否值得用

这个场景能帮助我记住：建议系统不是装饰，而是在做“能力导航”。

## 这里最值得记住的点

- 建议系统最好贴近当前上下文，而不是只给静态帮助文档
- suggestions 需要和设置、权限、可用工具状态联动
- 它更像引导层，不是主代理的替代品
- 真正值得学的地方，不是“会不会提示”，而是怎样把“能力很多”转成“当前可用、可发现、可行动”

## 易错点

- 容易把 advisor 理解成主代理的替代品
- 容易把建议系统看成静态帮助文档，而不是运行时上下文引导
- 容易忽视“能力过多”本身就是产品问题，建议层正是在解决这个问题
