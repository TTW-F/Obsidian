# Claude Code Hooks、Telemetry 与产品化观测

## 这是什么

这篇笔记记录 Claude Code 怎样通过 Hooks、Telemetry 和产品化观测建立运行时可见性。

它们合起来解决的是一个很实际的问题：系统跑起来之后，团队能不能看清它做了什么、哪里慢、哪里容易错，以及哪些能力真的被使用。

## 为什么重要

- agent 系统一旦进入真实使用，很快就会遇到性能、错误、卡点和能力利用率问题
- 如果没有观测层，系统虽然能运行，但会变成难排查、难优化、难运营的黑盒
- 对 Claude Code 这种带工具调用、长上下文、权限判断和扩展机制的产品来说，观测不是附属能力，而是运行基础设施的一部分

## 这三层各自更像在做什么

### Hooks

Hooks 更像正式的运行时插桩点。

它们适合承接那些跨多个运行节点、但又不该硬塞进主执行主线的横切逻辑，例如：

- session start
- prompt submit
- tool use
- compact 前后
- stop hooks

### Telemetry

Telemetry 更像把关键事件、性能和行为持续沉淀成可分析数据。

它关心的通常不是某个功能点，而是整条运行链路的状态，例如：

- 哪一步慢了
- 哪个工具老出错
- 用户在哪些流程里卡住
- 哪些能力实际被使用

### 产品化观测

产品化观测关注的是这些数据能不能真正反过来服务诊断、优化和迭代，而不只是留下一堆日志或埋点。

## Hooks 为什么不能只当日志回调

可以先把 Hooks 理解成“运行时里稳定的插桩位置”。

这样主循环负责推进任务，而 Hooks 负责把治理、观测、扩展和诊断所需的横切逻辑挂进来。两者分开之后，主链路不会因为杂项能力不断膨胀。

## Telemetry 为什么不是简单统计

对 Claude Code 来说，Telemetry 不只是统计“某按钮点了几次”，而是在回答：

- 性能怎样
- 工具链路怎样
- 插件和技能是否真正被使用
- 会话运行中发生了什么
- 某次异常是局部波动还是系统性问题

只有持续沉淀这些信息，团队才可能做性能分析、问题定位、体验改进和能力取舍。

## 一个具体场景怎么理解这层

如果某个工具在长会话里经常触发失败，单次日志可能只告诉你“这次报错了”，但产品化观测应该还能继续回答：

- 这个问题是不是集中发生在某个流程节点
- 是否和 context compaction、权限审批或特定插件有关
- 是偶发异常，还是最近版本才出现的趋势问题

这个场景能帮助我记住，Hooks 和 Telemetry 的价值，不在“记录一次事件”，而在“让系统行为可被长期理解”。

## 易错点

- 容易把 hooks 当成日志回调
- 容易把 telemetry 只看成产品统计
- 容易等系统复杂了再补观测层
- 容易只在 UI 层埋点，而忽略 session、tool use、context compaction、权限决策这些真正关键的运行节点

- 不要把这层价值缩成“多打了多少日志”。
  对复杂 agent 产品来说，能否被持续诊断和迭代，往往和能否跑通一次请求一样重要
