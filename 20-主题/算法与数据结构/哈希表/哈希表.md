# 哈希表

## 这是什么

哈希表可以先理解成：把“我要查什么”映射到一个更容易快速定位的位置上。

在做题时，我通常不先去想底层实现细节，而是先问：

- 我是不是需要快速判断某个值是否存在
- 我是不是需要记录某个值出现了几次
- 我是不是需要把一个对象映射到另一个信息

如果答案是“是”，哈希表往往就值得优先考虑。

## 为什么重要

- 它常常能把暴力枚举里的重复查找压缩掉
- 它是做去重、计数、映射时最常见的数据结构
- 很多初看像数组题、字符串题的问题，真正的突破口其实是哈希表

## 它到底在保存什么

我现在更习惯把哈希表看成“键到信息的映射”，而不是一个抽象名词。

常见的几种保存方式是：

- 值 -> 是否出现过
- 值 -> 出现次数
- 值 -> 上一次出现的位置
- 一个对象 -> 另一个对象

只要把“键”和“值”想清楚，很多题就会一下子清楚很多。

## 最常见的三种用途

### 1. 查存在

这是最直观的一类。

例如：

- 某个值之前见过吗
- 目标值的补数是否已经出现

这类题最重要的收益是把“每次都重新找一遍”变成“直接查一次”。

### 2. 计数

当题目关心频次时，哈希表特别合适。

例如：

- 某个字符出现了几次
- 两个字符串的字符分布是否一致
- 某个元素是否超过次数阈值

很多题看起来是在处理字符串或数组，其实是在维护频次表。

### 3. 建映射关系

有些题不是简单查存在，而是需要保存关系。

例如：

- 数值到下标
- 用户到状态
- 字符到最后出现位置

这类题里，哈希表真正提供的是“快速取回上下文”。

## 一个最常见的例子

比如两数之和：

如果暴力做法是双重循环，那么每次都在问：

“后面有没有一个数等于 `target - nums[i]`？”

哈希表的思路则是：

1. 一边遍历
2. 一边记录已经出现过的数及其下标
3. 当前看到一个数时，先查它的补数在不在

```python
seen = {}
for i, x in enumerate(nums):
    need = target - x
    if need in seen:
        return [seen[need], i]
    seen[x] = i
```

这段代码最该记住的不是模板，而是：

哈希表把“找补数”这件事从一段循环，变成了一次查表。

## 哈希表不是“所有查找问题”的万能答案

它确实快，但也有边界：

- 它通常需要额外空间
- 它不天然保留顺序结构
- 如果题目核心是连续区间、窗口约束或单调关系，别的方法可能更合适

所以我不会因为题里有“查找”两个字，就直接上哈希表。

## 和数组思维的一个关键差别

数组更强调位置、顺序和连续区间。

哈希表更强调“通过键快速拿信息”。

所以如果题目里最有价值的信息不是“第几个”，而是“这个值有没有、出现了几次、对应什么”，我就会更自然地想到哈希表。

## 易错点

- 容易只记 API，不知道当前哈希表到底在记录什么信息
- 容易把需要顺序的题也粗暴改成哈希表，结果丢掉结构
- 容易在更新时机上出错，比如应该先查再写，还是先写再查
- 容易把“哈希表能做”误当成“只有哈希表能做”

## 我的理解

哈希表最值得记住的不是“查找快”这句口号，而是：

它把值、状态和上下文建立成一张映射表，让重复查找变成一次直接访问。

只要题目核心在“查存在、查频次、查映射”，它通常就是非常自然的工具。
