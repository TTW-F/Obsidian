# 字符串

## 这是什么

字符串可以先理解成按顺序排在一起的一串字符。

但做题时，我不会只把它看成“文本”，而更倾向于把它看成一种特殊的数据序列。

因为很多字符串题真正考的不是语文意义上的文字，而是：

- 顺序关系
- 连续区间
- 字符统计
- 模式匹配

## 为什么重要

- 字符串是算法题里非常高频的输入形态
- 它经常把数组、哈希表、双指针、滑动窗口这些方法串到一起
- 如果这层理解不稳，就很容易把字符串题做成零散 API 记忆

## 字符串题常见的几种本质

### 1. 它其实是数组题

当题目关心的是位置、遍历、连续区间时，字符串本质上和数组很像。

例如：

- 反转字符串
- 找某一段子串
- 维护左右边界

这类题里，最重要的是顺序和区间。

### 2. 它其实是哈希表题

当题目关心的是字符出现次数、是否重复、上一次出现位置时，核心往往是哈希表。

例如：

- 判断字母异位词
- 最长无重复子串
- 统计字符频次

这类题不是真的在“研究字符串”，而是在维护字符到状态的映射。

### 3. 它其实是匹配问题

还有一类题的核心是：

- 某个模式能不能对上
- 两个序列的相对位置怎么比较
- 怎么避免重复匹配

这时就会走向 KMP、子串匹配、前后缀等更专门的内容。

## 字符串为什么容易让人误判

因为它太贴近日常语言了。

一看到字符串，很多人会先想语言函数和切片操作。

但做题时更重要的是先判断：

- 是连续区间问题吗
- 是计数问题吗
- 是模式匹配问题吗
- 是回文和对称问题吗

一旦这个判断做对，后面的工具选择会清楚很多。

## 一个常见边界

字符串题特别容易在这些地方出错：

- 空字符串
- 单字符
- 大小写是否区分
- 重复字符怎么处理
- 截取区间是左闭右开还是两端都闭

所以我现在会把“边界规范”看得和方法本身一样重要。

## 易错点

- 容易把字符串题做成 API 题
- 容易忽略它和数组、哈希表、窗口之间的联系
- 容易在切片和下标边界上出错
- 容易没先判断题目的核心结构就直接写代码

## 我的理解

字符串最值得记住的不是“字符序列”这个定义，而是：

它经常只是问题的表面载体，真正的解法结构可能来自数组、哈希表、双指针或匹配算法。

只要先看清这层结构，字符串题就不会显得那么散。
