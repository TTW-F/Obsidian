# 栈与队列

## 这是什么

栈和队列都可以先理解成“有顺序限制的线性结构”。

它们和数组、链表的差别不在于能不能存一串元素，而在于你取元素的顺序被刻意限制了。

- 栈是后进先出
- 队列是先进先出

## 为什么重要

- 很多题的核心不是查找快不快，而是处理顺序对不对
- 栈很适合表达“最近一次状态最重要”
- 队列很适合表达“按进入顺序逐个处理”

## 栈最该怎么理解

我现在更倾向于把栈理解成“离当前最近的历史状态”。

只要题目里有这种感觉，栈就会很自然：

- 最近一个还没配对的括号
- 最近一个更大 / 更小关系还没确定的元素
- 最近一次进入但还没退出的状态

这也是为什么括号匹配、表达式处理、单调栈都和它很近。

## 队列最该怎么理解

队列更像“排队等待被处理的对象”。

只要题目里是这种感觉，队列通常就值得试：

- 先来的先处理
- 一层一层往外扩
- 当前加入的任务稍后再处理

所以层序遍历、BFS、任务调度这些问题，经常天然带着队列味道。

## 一个最常见的栈场景

比如括号匹配：

每遇到一个左括号，就先压入栈；
每遇到一个右括号，就尝试和栈顶最近的左括号配对。

这个例子最能说明栈的本质：

题目关心的不是所有历史，而是“最近那个还没处理完的状态”。

## 一个最常见的队列场景

比如二叉树层序遍历：

1. 先把根节点放进队列
2. 每次取出队头节点
3. 再把它的左右孩子按顺序加入队尾

这里最重要的不是代码，而是队列自然地维持了“按层、按进入顺序处理”的节奏。

## 它们和数组、链表的关系

数组和链表更像底层承载结构。

栈和队列更像“使用规则”。

也就是说，你可以用数组或链表去实现它们，但做题时最重要的不是底层实现，而是：

当前问题是不是需要这种受限顺序。

## 易错点

- 容易把栈和队列当成 API 题，而不是顺序约束题
- 容易在明明需要最近状态时，还去用普通遍历硬做
- 容易记住“后进先出 / 先进先出”，却不知道题里谁该进、谁该出
- 容易忽略队列在 BFS 和层序问题里的自然性

## 我的理解

栈与队列最值得记住的不是定义，而是：

它们把“元素怎么被处理”这件事强行变得有纪律。

一旦题目核心是处理顺序，栈和队列就往往比普通遍历更贴题。
