# 动态规划

## 这是什么

动态规划可以先理解成：把一个大问题拆成许多会重复出现的小问题，并把这些小问题的答案存下来，避免重复计算。

所以它不是“公式技巧”，而是一种组织子问题的方法。

## 为什么重要

- 很多暴力递归之所以慢，就是因为同样的子问题被反复算了很多次
- 动态规划能把“指数级乱试”压成“按状态有序计算”
- 很多最优解、方案数、可达性问题都能在这里汇合

## 动态规划最该先想什么

我现在做 DP 题时，最先逼自己写清三件事：

1. 状态是什么
2. 状态转移怎么来
3. 初始状态和边界是什么

如果第一件事没写清，后面两个通常都会飘。

## 状态到底是什么

状态不是数组名，也不是循环下标。

状态真正表达的是：

“我把问题做到某个阶段时，手里保存的这个量，具体代表什么。”

例如：

- `dp[i]` 表示到第 `i` 个位置时的最优值
- `dp[i][j]` 表示考虑到前 `i` 个物品、容量为 `j` 时的最优值

所以我现在最重视的不是先写式子，而是先把 `dp[...]` 用一句人话说清楚。

## 为什么很多人觉得 DP 难

因为它表面上像在写数组，实际上是在定义子问题。

一旦状态含义模糊，后面就会出现这些问题：

- 转移为什么这样写说不清
- 初始化全靠猜
- 代码能跑，但不知道是不是对的

所以 DP 难的地方通常不在代码，而在建模。

## 一个最常见的例子

比如爬楼梯：

如果走到第 `i` 阶，只可能从第 `i-1` 阶或第 `i-2` 阶过来，那么：

`dp[i] = dp[i - 1] + dp[i - 2]`

这个例子很经典，但我现在更看重的是它背后的结构：

当前问题的答案，可以由更小规模的同类问题组合出来。

这就是 DP 最核心的味道。

## 动态规划和回溯的区别

回溯会把很多路都试一遍。

动态规划更像是在说：

既然这些子问题会反复出现，那我不如把答案记下来，下次直接用。

所以如果我看到“重复子问题”很明显，就会优先警觉 DP。

## 动态规划和贪心的区别

如果一个问题能在每一步都做不会后悔的选择，贪心可能更简单。

但如果当前这一步到底该怎么选，必须看多种未来情况，DP 往往更稳。

我现在会这样理解：

- 贪心想一步定局
- DP 承认要把很多可能状态都保留下来比较

## 易错点

- 容易先背公式，后补状态定义
- 容易把初始化写成经验值，而不是由状态定义推出
- 容易在遍历顺序上出错，导致转移依赖还没准备好
- 容易分不清这题是回溯、贪心还是 DP

## 我的理解

动态规划最值得记住的不是 `dp` 这个名字，而是：

先定义清楚“这个状态到底表示什么”，再让更小问题的答案有组织地推出来。

只要状态定义站稳，很多 DP 题其实是在同一种思路下换壳。
