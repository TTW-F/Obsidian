# Agentic CLI 研究路线

## 这是什么

这篇笔记记录的是一条适合反复复用的研究顺序：当我拿到一个新的 agentic CLI 产品或源码仓库时，应该先看哪几层，才能更快建立稳定心智模型。

它不是某个产品的读后感，而是一张通用阅读路线图。

## 为什么重要

- 这类系统的概念很多，如果没有顺序，很容易只记住零散功能名词
- 先抓系统骨架，再看局部模块，后续读源码、文档和示例会更容易对齐
- 同一条路线可以复用到不同终端 agent 工具上

## 推荐研究顺序

### 1. 先看任务入口

先确认任务从哪里进入系统，例如 CLI 参数、交互 REPL、slash command、IDE bridge。

这里要回答的问题是：用户表达一个任务时，系统有没有统一入口，还是存在多套彼此割裂的调用路径。

### 2. 再看执行主循环

这是最关键的一层，要搞清楚：

- 用户输入怎样进入推理
- 工具结果怎样回到下一轮推理
- 一轮何时结束，任务何时继续
- 会话状态怎样保存和恢复

### 3. 再看能力层

这一层关注系统能调哪些能力，以及能力是怎样声明的。常见对象包括 tools、commands、文件系统、Shell、Web、搜索、MCP、LSP、Task 和子代理。

### 4. 再看治理层

这里重点看权限、策略、设置、日志、遥测、预算和持久化。很多“看起来能跑”的 agent，真正的问题就出在这层不完整。

### 5. 最后看扩展层

最后再看 skills、plugins、MCP 和多代理机制，判断系统是单体实现，还是已经具备持续演化的结构。

## 怎么使用这条路线

- 看新产品时，先用它判断骨架，再决定要不要深挖
- 读源码时，按顺序找目录和关键文件，避免一上来扎进局部实现
- 读官方文档或示例时，把每部分映射回“入口、主循环、能力、治理、扩展”这五层

## 易错点

- 容易先看工具清单，结果对系统如何推进任务没有整体把握
- 容易把扩展机制当成最后才需要关心的细节，实际上它会反过来暴露架构是否清晰
- 容易把阅读顺序写成一次性的研究计划，而不是能长期复用的方法

- 研究路线的价值，是帮助我先抓系统骨架，再决定哪些局部模块值得深挖
