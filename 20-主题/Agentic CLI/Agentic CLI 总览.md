# Agentic CLI 总览

## 这是什么

`Agentic CLI` 可以先理解成一种把大模型、工具调用、会话状态和权限控制放进终端里的执行型系统。

它和普通聊天机器人最大的区别，不是界面像命令行，而是模型在回答之外还能调用文件、Shell、搜索、Web、MCP 等能力，持续推进一个任务。

## 为什么重要

- 它把“问答工具”推进成“任务执行工具”，更适合开发、运维、研究和自动化场景
- 它要求系统同时解决执行力、权限边界、状态延续和可恢复性，工程复杂度明显高于单轮对话
- 它是理解现代 AI coding agent、终端助手和 IDE agent 的共同入口

## 先从哪几层理解

### 1. 用户入口

用户怎么把任务交给系统，包括 CLI 参数、交互 REPL、斜杠命令和 IDE Bridge。

### 2. 执行主线

系统怎样把一轮用户输入变成多轮推理与行动，包括上下文拼接、模型调用、工具执行、结果回流和终止条件。

### 3. 能力层

模型到底能做什么，通常落在文件系统、Shell、搜索、Web、代码编辑、任务拆分、子代理和外部协议接入上。

### 4. 治理层

系统如何限制风险，包括权限检查、策略配置、日志、遥测、预算和持久化。

### 5. 扩展层

系统如何在不改核心代码的前提下继续长大，例如 skills、plugins、MCP、hooks 和团队级扩展机制。

## 判断一个 Agentic CLI 是否成熟时先看什么

可以先看下面五件事：

1. 任务入口是否统一，用户输入能否稳定落到同一条执行主线
2. 主循环是否清楚，模型调用、工具执行和状态更新有没有稳定闭环
3. 工具系统是否可治理，能力声明、权限控制和结果注入是否一致
4. 长任务是否可恢复，历史压缩、状态续跑和失败回退是否被认真处理
5. 扩展机制是否成体系，新能力接入后能不能继续受同一套规则约束

## 易错点

- 容易把“能调很多工具”误解成 agent 已经成熟，真正的难点通常在治理和恢复能力
- 容易只盯工具实现，忽略任务是怎样被一轮轮推进的
- 容易把某个产品的实现细节当成通用规律，实际要区分产品选择和可迁移模式

## 最该记住的点

- Agentic CLI 的重点不在“会不会回答”，而在它是否能把执行、治理、状态和扩展放进同一套运行结构
