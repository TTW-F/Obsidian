# 工具调用系统

## 这是什么

工具调用系统解决的问题是：怎样把“模型能做什么”描述成一组结构化能力，再把这些能力安全地接到真实执行环境上。

在 agentic CLI 里，工具系统通常是执行力的核心，因为模型只有能稳定调用文件、Shell、搜索、Web、外部协议或协作能力，才真正像在做事。

## 为什么重要

- 没有工具，模型更多是在建议；有了工具，模型才能真正操作环境
- 工具系统的设计会直接决定 agent 是演示型产品，还是可长期使用的生产工具
- 工具调用还会牵出权限、结果格式、错误处理和可观测性这些后续问题

## 成熟工具系统通常要回答哪些问题

### 1. 工具怎么注册和描述

工具描述要足够清楚，让模型知道什么时候该选它、输入应该长什么样。

### 2. 工具和命令怎么分层

很多系统里，命令更偏人类显式入口，工具更偏模型可调用能力。这两层如果混在一起，交互和执行边界就会很乱。

### 3. 输入输出是否可治理

工具输入是否可校验，输出是否能被下一轮稳定消费，通常决定了工具系统能不能规模化。

### 4. 权限和模式怎么接进来

同一个工具在不同模式下，可能要被允许、拦截、审批或裁剪。

## 工具系统里常见的分层方式

- `commands` 面向人，代表显式入口
- `tools` 面向模型，代表可调用能力
- 工具协议层通常会定义输入 schema、运行上下文、权限上下文、进度事件和状态回写能力

这说明清楚的工具系统不会把“命令”和“工具”混成一层。

## 常见工具类别

- Shell / PowerShell
- 文件读取、编辑、写入
- 搜索：glob、grep、web search、web fetch
- 外部协议：MCP、LSP
- 协作：Agent、SendMessage、TeamCreate
- 任务：TaskCreate、TaskUpdate、TaskList

## 一个具体场景怎么理解工具系统

比如用户要求“在仓库里找某类文件、修改一段代码、再验证结果”。

这时 agent 往往不会只调用一个工具，而是会连续经过：

1. 搜索工具找到候选文件
2. 读取工具拿到上下文
3. 编辑工具改动内容
4. Shell 或测试工具验证结果

所以工具系统真正重要的，不是某个工具单独多强，而是这些结果能不能继续被下一轮稳定消费。

## 易错点

- 容易把工具系统理解成“工具越多越好”，忽略边界、粒度和可组合性
- 容易把命令和工具混写，导致用户入口和模型能力边界不清
- 容易只关心能不能调用，不关心输出是否适合下一轮继续消费

- 好的工具系统不只是把功能列出来，还要让工具能力、结果结构和权限治理能一起工作
