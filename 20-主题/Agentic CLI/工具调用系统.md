# 工具调用系统

## 这是什么

工具调用系统，就是把“模型能做什么”描述成一组结构化能力，再把这些能力安全地接到真实执行环境上。

这里重点记录的是从 `Claude Code` 里观察到的工具系统设计，不展开成泛化的 agent framework 教科书。

## 为什么重要

- 没有工具，模型只能建议
- 有了工具，模型才能搜索、读写文件、执行命令、调用外部系统
- 工具系统的设计，直接决定 agent 是玩具还是生产工具

## 研究这类系统时我会先问什么

- 工具是怎么注册的
- 工具和命令有没有分层
- 工具输入能不能校验
- 工具结果是否适合被下一轮继续消费
- 工具能力能不能按权限模式裁剪

## Claude Code 给我的启发

在 Claude Code 里，`commands.ts` 和 `tools.ts` 是两条不同的线：

- `commands` 面向人，代表显式入口
- `tools` 面向模型，代表可调用能力

`Tool.ts` 进一步定义了：

- 工具输入 schema
- 工具运行上下文
- 权限上下文
- 进度事件
- UI / 状态回写能力

这说明成熟系统不会把“命令”和“工具”混成一层。

## 典型工具类别

- Shell / PowerShell
- 文件读取、编辑、写入
- 搜索：glob、grep、web search、web fetch
- 外部协议：MCP、LSP
- 协作：Agent、SendMessage、TeamCreate
- 任务：TaskCreate、TaskUpdate、TaskList

## 设计时要关注什么

- 工具描述是否足够清晰，让模型会选、会用
- 输入 schema 是否可验证
- 返回结果是否便于下一轮推理继续消费
- 工具执行是否能暴露进度、错误和中间状态
- 工具是否能在不同模式下被裁剪、禁用或延迟加载

## 我的理解

好的工具系统不是“工具越多越好”，而是：

- 边界清楚
- 粒度合适
- 结果可组合
- 权限可治理
