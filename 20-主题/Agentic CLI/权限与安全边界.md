# 权限与安全边界

## 这是什么

权限系统决定 agent 哪些事可以自动做，哪些事必须拦住、询问、拒绝。

这里重点保留 `Claude Code` 这类编码代理里的权限治理经验，避免和更抽象的安全框架笔记重复。

## 为什么重要

- 一旦接入 Shell、文件写入、网络访问，agent 就进入高风险区
- 真正可用的 agent 不是“什么都能做”，而是“在边界内稳定做事”
- 权限系统本质上是 agent 的治理层

## 研究这部分时我会先看什么

- 权限是按工具、按命令，还是按更细的规则控制
- allow / deny 规则能不能落到具体路径、前缀、模式
- 自动执行模式和人工确认模式怎么切换
- 子代理权限是否会偷偷放大风险

## Claude Code 里值得学的点

`utils/permissions/permissionSetup.ts` 不只是做开关，它还在处理：

- 权限模式切换
- 规则加载与解析
- 危险规则识别
- auto mode 下的安全收敛
- Bash / PowerShell 的高风险前缀规则识别
- 子代理权限风险控制

这说明权限系统不是 UI 弹窗，而是一套策略引擎。

## 我观察到的几个关键思想

- 工具是否暴露给模型，和调用时是否放行，是两层判断
- 某些“过于宽泛”的 allow 规则本身就是风险
- PowerShell / Bash 不是一个抽象命令，而是一整类高危执行能力
- 子代理自动批准同样危险，因为它会绕过对 prompt 的安全判断

## 设计原则

- 默认保守，按需放行
- 规则要能落到具体工具、具体前缀、具体目录
- 高危工具要做专门分类，不能只靠通用 allow / deny
- 模式切换要可恢复，避免进入 plan / auto 后状态混乱

## 我的理解

权限设计真正保护的，不只是机器，也是在保护使用者的“决策节奏”。

如果边界过松，系统会失控。
如果边界过紧，系统就失去执行力。

所以成熟产品拼的是“有弹性的约束”。
