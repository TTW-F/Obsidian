# 权限与安全边界

## 这是什么

权限与安全边界讨论的是：agent 哪些事可以自动做，哪些事必须拦住、询问、审批或拒绝，以及这些规则怎样真正落到工具和执行环境上。

在 agentic CLI 里，这不是附属功能，而是执行力能否安全释放出来的前提。

## 为什么重要

- 一旦接入 Shell、文件写入、网络访问和外部服务，agent 就进入高风险区
- 真正可用的 agent 不是“什么都能做”，而是“在边界内稳定做事”
- 权限系统会直接影响用户信任、产品可用性和长任务执行能力

## 权限系统真正要回答哪些问题

### 1. 哪些能力先天高风险

Shell、文件写入、网络访问、远程连接和子代理自动执行，通常都比单纯读取信息风险更高。

### 2. 规则能不能落到具体对象

清楚的权限系统不会只停留在“允许 / 不允许”这种粗粒度开关，还会落到具体工具、具体路径、具体前缀或具体模式上。

### 3. 自动执行和人工确认怎么切换

有些操作可以自动放行，有些操作需要显式确认，还有些操作应该直接拒绝。关键在于切换规则是否清楚、是否可恢复。

### 4. 子代理会不会放大风险

多代理系统里，权限问题会更复杂，因为风险不只来自工具本身，也来自“谁可以代替我继续执行”。

## 权限系统常见会落在哪些位置

如果一个系统把权限当成策略引擎而不是单次确认框，常见会集中处理：

- 权限模式切换
- 规则加载与解析
- 危险规则识别
- auto mode 下的安全收敛
- Bash / PowerShell 的高风险前缀识别
- 子代理权限风险控制

这说明权限系统更像策略引擎，而不是单次确认框。

## 一个具体场景怎么理解权限边界

如果 agent 需要在仓库里运行 shell 命令并修改文件，系统至少要回答：

1. 当前模式是否允许执行 shell
2. 这个命令前缀是否属于高风险动作
3. 写文件是否只允许落在特定目录
4. 如果由子代理执行，是否还要重新检查边界

这个场景说明，权限问题不是“最后弹个窗”就解决了，而是贯穿执行链。

## 更适合先记住的几个原则

- 工具是否暴露给模型，和调用时是否放行，是两层判断
- 允许规则太宽，本身就是风险
- 高危能力不能只靠通用 allow / deny 处理，通常要有专门策略
- 多代理环境里，权限不能默认被继承和放大

## 易错点

- 容易把权限系统理解成 UI 弹窗，而忽略它本质上是运行时治理层
- 容易把所有工具放进同一套粗粒度规则，结果高危能力缺少专门约束
- 容易只盯当前代理，忽略子代理和后台执行也会持续放大风险

- 权限设计既要拦住高风险动作，也要避免把系统卡成只能建议、不能执行
