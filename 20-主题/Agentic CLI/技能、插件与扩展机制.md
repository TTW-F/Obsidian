# 技能、插件与扩展机制

## 这是什么

扩展机制讨论的是：当 agent 的内建能力不够时，系统怎样新增能力，同时又不把整体架构搞乱。

在 agentic CLI 里，常见扩展对象包括技能、插件、MCP 服务、外部资源和 hooks。它们共同解决的不是“多几个功能”，而是“怎样可持续地接入新能力”。

## 为什么重要

- 用户需求变化很快，很多能力不可能全部硬编码进核心系统
- 如果扩展没有统一治理，新增能力越多，系统越容易失控
- 生态上限往往不是由模型决定，而是由扩展能力的接入成本和治理质量决定

## 可以从哪几个维度理解扩展机制

### 1. 扩展单元长什么样

一个扩展是否有稳定元数据、入口约定和配置结构，决定了它能不能被发现、校验和复用。

### 2. 扩展怎样被发现和加载

要看系统有没有完整的发现、加载、版本管理、启动检查和缓存链路。

### 3. 扩展能力是否仍受治理

扩展能力不能绕开权限、策略和审计系统，否则扩展越强，风险越大。

### 4. 扩展层次是否清楚

官方扩展、项目扩展和用户扩展最好能分层共存，否则覆盖关系和维护边界会很混乱。

## 先把几种扩展区分开

可以先这样区分：

- `skill`：更轻量，适合沉淀工作流、知识和约束
- `plugin`：更正式，适合打包、分发、安装和版本管理
- `MCP`：更偏协议接入，适合把系统接到外部工具和资源

它们都在扩展能力，但解决的问题层次并不一样。

## 这几类扩展通常分别在补什么

- `skill` 更像带元数据的能力单元，而不只是提示词片段
- `plugin` 更偏打包、版本、安装和分发
- `MCP` 更偏通过协议接入外部工具和资源

比这些分类更重要的是背后的共同要求：

- 能被统一发现
- 能被统一治理
- 能被统一接入

## 判断扩展机制是否成熟时可以看什么

1. 扩展元数据是否统一，是否能描述用途、权限和依赖
2. 发现、加载、校验、缓存和升级是否形成完整链路
3. 扩展接入后是否仍然受权限与策略控制
4. 用户扩展、项目扩展、官方扩展之间是否有清楚边界

## 易错点

- 容易把 skill、plugin、MCP 当成互相替代的名词，实际上它们解决的问题层次不同
- 容易只记录功能点，不记录加载和治理链路，结果后续很难复用判断标准
- 容易把某个产品的扩展命名当成普遍分类，实际需要先看它在架构里承担什么职责

- 扩展机制真正难的不是多加入口，而是新能力进来后还能不能继续遵守原有治理规则
