# 技能、插件与扩展机制

## 这是什么

扩展机制解决的问题是：当内建能力不够时，系统如何持续长大，而且不会越长越乱。

这里默认从 `Claude Code` 的 skill / plugin / MCP 实现切入，只提炼能被迁移的方法，不抢占通用框架笔记的位置。

## 为什么重要

- 真正可长期演化的 agent，不能只靠硬编码
- 用户需求变化很快，扩展能力必须低成本接入
- 扩展系统决定生态上限

## 研究这部分时我会先看什么

- 扩展单元有没有统一元数据
- 扩展的发现、加载、校验链路是否完整
- 扩展能力是否仍受权限治理
- 官方扩展、项目扩展、用户扩展是否能分层共存

## Claude Code 里我看到的三条扩展线

### 1. Skills

`skills/loadSkillsDir.ts` 显示 skill 本质上是带 frontmatter 的 Markdown 配置对象。

它可以声明：

- 描述
- 何时使用
- allowed tools
- 参数
- model / effort
- hooks
- 执行上下文

这让我意识到 skill 不只是“提示词片段”，而是半结构化能力单元。

### 2. Plugins

`plugins/` 与 `utils/plugins/` 说明插件系统已经覆盖：

- 加载
- 版本管理
- 启动检查
- 市场与官方源
- 缓存
- 命令 / 技能 / hooks / output style 扩展

### 3. MCP / 外部资源

MCP 让系统不必把所有能力硬写进本体，而是通过协议接入外部工具和资源。

## 我提炼出的几个原则

- 扩展单元要有统一元数据
- 扩展发现、加载、校验、缓存要是完整链路
- 扩展不能绕过权限与策略系统
- 用户扩展、项目扩展、托管扩展要能分层覆盖

## 我的理解

成熟的 agent 产品不是靠“更多功能”取胜，而是靠“把新增功能纳入同一套治理框架”取胜。
