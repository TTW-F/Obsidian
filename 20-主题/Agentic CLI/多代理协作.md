# 多代理协作

## 这是什么

多代理协作，就是把一个大任务拆给多个子代理并行处理，再由主代理整合结果。

这篇主要关注 `Claude Code` 源码里已经落地的代理配置与协作机制，而不是泛讲所有多 agent 理论。

## 为什么重要

- 大任务里经常存在可并行的子问题
- 单代理上下文有限，容易把“分析”“执行”“验证”混在一起
- 多代理可以提升吞吐，也能形成职责分离

## 研究这部分时我会先问什么

- 子代理是临时任务，还是正式资源
- 能不能为不同代理配置不同工具边界
- 上下文怎么继承，怎么隔离
- 结果怎么回收，怎么整合

## Claude Code 里给我的启发

从目录上就能看到它不是简单的“spawn 一个子任务”：

- `tools/AgentTool/`
- `coordinator/`
- `utils/swarm/`
- `SendMessageTool`
- `TeamCreateTool` / `TeamDeleteTool`

再看 `loadAgentsDir.ts`，可以发现 agent 本身也被当成可配置对象：

- 有 agent type
- 有 system prompt
- 有 allowed / disallowed tools
- 有 permission mode
- 有 model / effort
- 有 memory scope
- 有 isolation 模式
- 还能挂 MCP、hooks、skills

这意味着“代理”在系统里已经是一级资源。

## 多代理系统的关键问题

- 怎么拆任务，避免互相阻塞
- 怎么定义每个代理的工具边界
- 怎么同步权限与上下文
- 怎么回收、整合、复用结果
- 怎么避免代理之间覆盖彼此工作

## 我提炼出的方法

- 把主代理视为编排器，而不是万能执行者
- 把子代理视为带角色约束的可替换执行单元
- 把协作机制视为协议问题，而不是单纯并发问题

## 我的理解

多代理不是“多开几个模型”。

它本质上是在设计一套组织结构，让不同能力单元在共享目标下低冲突协作。
