# 多代理协作

## 这是什么

多代理协作可以先理解成：把一个大任务拆成多个相对独立的工作单元，交给不同代理并行或分阶段处理，再把结果回收到同一条任务主线上。

在 agentic CLI 场景里，它讨论的不只是“多开几个模型”，而是角色分工、工具边界、上下文隔离和结果回收这些组织问题。

## 为什么重要

- 大任务里常常同时存在分析、执行、验证、整理等不同类型的子工作
- 单代理容易把所有职责揉在一起，既占上下文，又容易互相干扰
- 多代理如果设计得好，可以提升吞吐，也能让职责边界更清楚

## 多代理系统真正要解决什么

### 1. 任务怎么拆

不是所有任务都适合并行。真正适合拆出去的，通常是边界清楚、结果格式明确、不会阻塞主线判断的子问题。

### 2. 角色怎么分

主代理更像编排者，子代理更像带角色约束的执行单元。常见分工包括：

- planner / coordinator
- researcher / explorer
- worker / implementer
- verifier / reviewer

### 3. 边界怎么控

不同代理未必要共享同一套工具、权限、模型和上下文。很多系统真正成熟的地方，就在于它们允许为不同角色裁剪能力。

### 4. 结果怎么回收

多代理系统不是把任务扔出去就结束了，还要回答结果如何汇总、冲突如何处理、哪些上下文需要带回主线。

## 多代理实现里常见会出现哪些配置项

如果一个系统把“代理”当正式运行单元，而不是临时 prompt 技巧，常见会给 agent 定义这些配置项：

- agent type
- system prompt
- allowed / disallowed tools
- permission mode
- model / effort
- memory scope
- isolation 模式

这说明多代理的关键不是“多实例”，而是“可配置的角色单元”。

## 一个具体场景怎么理解多代理

如果主代理收到一个较大的代码库任务，它可能会：

1. 自己保留总体目标和结果整合
2. 派一个子代理查找相关文件
3. 派另一个子代理做局部实现
4. 再派一个子代理做验证或 review
5. 最后由主代理汇总结果并决定下一步

这个场景里真正重要的，不是并发本身，而是每个代理的职责和边界都不同。

## 易错点

- 容易把多代理问题只看成并发问题，忽略角色分工、权限同步和结果回收
- 容易让子代理默认继承主代理全部能力，结果边界过宽
- 容易只关注“派出去做什么”，不设计“结果怎样带回来”

- 多代理协作讨论的不只是“多几个执行者”，而是怎样把任务拆分、角色裁剪、结果回收和边界控制组合成稳定流程
